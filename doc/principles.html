YADS Model Principles

In order to support resolution of a DOI to multiple resources we make use of the following principles:
Resources
Resources are either:
  • "Literals" (rdf:resource, mapping 1:1) - inline resources identified by URI of scheme "data:" (RFC 2397), ie literals are represented as resources.
  • "Resources" (rdf:resource, mapping 1:1) - familiar web resources identified by URI; properties: anonymous, maintained by a single organization or enterprise.
  • "Meta-Resources" (rdf:resource, mapping 1:N) - resources identified by URI manging resource collections; properties: named, maintained across a common business sector. A good example of such a resource is a DOI (URI scheme "doi:").
Hierarchy
There is a clear need to organize resources - by grouping, by hierarchy, by ordering.
Type Primitives
Resources are typed against a limited set of primitive data types - see Model Schema
Resource Map
A simple Resource Map construct allows for a resource hierarchy to be reposited in a flat DataStore. This Resource Map serves as a simple driver to generate a resource hierarchy serialization, eg an RDF/XML instance.
Transfer Syntax
The DOI resource description is serialized as an XML document. In particular an RDF/XML serialization provides the necessary constraints. This is purely an "under the hood" exchange syntax which is not intended to be exposed to DOI end users, although it does allow for downstream interop with other RDF aware applications. The DOI Client/Service interop stack is defined for both distributed and localized implementations.
Neutral Semantics
The semantics that are to be exchanged between a DOI client application and a DOI service are primarily structural. The client/service transactions are effectively dealing with a type of "Poorboy RDF".
Late Binding
An extensibility type mechanism is provided through the type "has:type". As an example consider how a new eBook format type could be introduced: 


<rdf:li rdf:parseType="Resource">
  <has:resource>data:,Microsoft%20Reader<has:resource>
  <has:type>ebook:Format</has:type>
</rdf:li>

Application specific semantics are recovered from an Application specific schema which is registered with the DOI Application Profile. This mechanism allows the generic RDF for a DOI to be specialized (or enriched) into an Application specific RDF: 


<rdf:li rdf:parseType="Resource">
  <ebook:Format>Microsoft Reader</ebook:Format>
</rdf:li>
